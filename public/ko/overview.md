## AX: AI 전환 — 핵심 역량으로서의 AI 활용

AI 활용은 선택이 아닌 **핵심 엔지니어링 역량** — 단일 개발자가 달성할 수 있는 것을 근본적으로 바꿉니다.

### Cline → Claude Code 전환

2025년 8월 이전에는 **Cline** (VS Code 확장)을 AI 코딩 보조 도구로 사용했습니다. 유용했지만 한계가 있었습니다 — 단일 파일 컨텍스트, 자율 실행 불가, 병렬 처리 불가.

**2025년 8월 16일**, **Claude Code** (CLI)를 도입했습니다. 첫 `CLAUDE.md`가 오후 1시 14분에 커밋되었고, 첫 Claude 보조 코드 커밋 — Expo 마이그레이션 1단계 — 이 11분 후에 생성되었습니다. 10일 이내에 모든 주요 레포지토리에 Claude Code 워크플로우를 배포했습니다:

|------|------|--------|

| 8월 19일 | legacy-service | Claude PR 리뷰 워크플로우 |
| 8월 25일 | MSA 백엔드 | Claude PR 리뷰 워크플로우 |

이후 팀 전체 개발자를 위해 **Max 20x 플랜**을 구매했습니다. 이것은 단순한 개인 생산성 도구가 아닌 — 팀 전체의 소프트웨어 배포 방식을 바꾼 **조직 차원의 AX(AI Transformation)** 결정이었습니다.

### 워크플로우 동작 방식

**Tmux 멀티 페인 오케스트레이션:**
- 4-8개 Claude Code 에이전트를 분할 터미널 패널에서 동시 실행
- 각 에이전트가 비중복 파일셋을 소유하여 충돌 방지
- 에이전트가 자율 작업 — 코드베이스 읽기, 변경 계획, 구현, 자체 검증

**야간 자율 실행:**
- 복잡한 작업을 큐에 추가 (타입 안전성 페이즈, 테스트 스위트, 리팩토링 캠페인)
- 에이전트가 밤새 연속 처리
- 아침에 리뷰 가능한 완성된 PR 확인

**페이즈 기반 병렬 처리:**
- 대규모 캠페인을 독립적 페이즈로 분해 (예: 50+ PR 타입 안전성 에픽)
- 동시 에이전트 세션에 페이즈 분산
- 공유 `CLAUDE.md` 지침과 이슈 트래킹으로 조율

이 방법론을 통해 단일 엔지니어가 5인 이상 팀에 필적하는 산출물 생산 — 12개월간 28개 레포지토리에 걸친 3,775건의 커밋과 1,000건 이상의 PR을 가능하게 한 직접적 원동력.

### AI 기반 엔지니어링을 성공시키는 원칙

도구보다 접근 방식이 중요합니다. AI를 코드 자동완성이 아닌 전력 증폭기로 전환한 원칙들:

**1. AI를 도구가 아닌 인프라로 다룬다.**
Claude Code를 설치하고 프롬프트만 입력한 것이 아닙니다. AI 주변에 **프로덕션 파이프라인**을 설계했습니다 — tmux 오케스트레이션, 레포별 `CLAUDE.md` 규칙, 야간 실행 큐, 페이즈 기반 작업 분해. 18배 산출량 증가는 더 나은 프롬프트가 아닌 AI를 둘러싼 시스템 엔지니어링에서 나왔습니다.

**2. 지시가 아닌 의도를 위임한다.**
효과적인 AI 위임은 단계별 지시가 아닌 **결과와 제약 조건**을 전달하는 것입니다. "테스트를 통과시키면서 admin 모듈의 모든 `any` 타입을 제거하라"는 작동합니다. "파일 X를 열어서 42번 줄을 수정하고..."는 확장되지 않습니다. 핵심 역량은 각 작업에 맞는 적절한 추상화 높이를 아는 것입니다.

**3. 속도가 아닌 조율을 설계한다.**
8개 AI 에이전트를 동시에 돌려도 충돌하면 무용지물입니다. 각 에이전트가 **비중복 파일셋**을 소유합니다. 대규모 캠페인은 독립적 페이즈로 분해합니다. 공유 `CLAUDE.md`로 드리프트를 방지하는 규칙을 확립합니다. 어려운 문제는 AI를 빠르게 만드는 것이 아니라 병렬 AI 작업을 **일관되게** 만드는 것입니다.

**4. 품질은 내가, 실행은 AI가.**
AI가 코드를 작성합니다. 아키텍처, 리뷰, 무엇을 만들지의 판단은 제가 합니다. 모든 PR을 리뷰합니다. 모든 캠페인은 사람이 정의한 범위에서 시작합니다. 야간 에이전트는 PR을 만들지 — 머지된 코드를 만들지 않습니다. 이 분리가 AI 산출물을 데모 수준이 아닌 프로덕션 수준으로 유지합니다.

**5. 개인이 아닌 팀으로 확장한다.**
모든 개발자에게 Max 20x 구매, 10일 내 7개 레포에 PR 리뷰 워크플로우 구축, 팀 누구나 따를 수 있는 `CLAUDE.md` 규칙 작성 — 이것은 개인 생산성 해킹이 아닌 **조직 전환**이었습니다. 목표는 "내가 더 빨리 배포한다"가 아니라 "팀이 더 빨리 배포한다"였습니다.

### 코드를 넘어서: AI 운영 시스템

위의 5가지 원칙은 *코딩*을 가속화합니다. 그러나 진정한 AX 프론티어는 AI가 자율적으로 운영하는 시스템을 구축하는 것 — 코드를 작성하는 것이 아닌, 프로덕션 운영을 수행하는 것입니다.

**[자율 Sentry Issue 분석 및 PR 생성 봇](projects/sentry-triage-bot.md)**이 가장 명확한 사례: Sentry 웹훅이 4개 레포지토리에 걸친 AI 기반 이슈 분석을 트리거하고, 환경 인식 정책 게이트, 읽기 전용 DB 검증, 프로덕션 이슈 자동 PR 생성, 텔레그램 리포팅까지 — 사람의 개입 없이 모두 수행하는 파이프라인입니다.

알림에서 수정 PR까지 수시간이 아닌 수분. 온콜 엔지니어 불필요.

이것은 "AI가 코딩을 빠르게 도와준다"에서 "AI가 전담 온콜 엔지니어가 필요했던 프로덕션 운영을 처리한다"로의 진화를 의미합니다. Sentry 봇은 5가지 원칙을 단순히 사용하는 것이 아닌 *체현*합니다: 인프라(도구가 아닌)이고, 의도를 수신(지시가 아닌)하고, 레포 간 조율(속도만이 아닌)하고, 사람의 리뷰를 위한 PR을 생산(품질 소유)하며, 팀 전체를 위해 운영(개인이 아닌)됩니다.

## 복리 성장의 아크: 플랫폼을 어떻게 재구축했는가

2025년 8월 CTO가 퇴사했을 때, 물려받은 것은 취약한 플랫폼이었습니다: 공유 인프라 없는 3개의 분산된 프론트엔드 레포, 검색엔진에 보이지 않는 정적 웹앱, 제로 테스트 커버리지, 에러 모니터링 부재, 환경 변수가 하드코딩된 29개 Lambda 백엔드. 화재 진압식 대응으로도 서비스는 유지할 수 있었습니다. 저는 순차적 투자를 선택했습니다 — 각 결정이 의도적으로 다음 결정을 가능하게 하도록.

**현대화 → 통합 → 안정화 → 최적화 → 자동화**

1. **기반 현대화** (9~10월): 웹 플랫폼을 Next.js 15로 업그레이드하고 GCP Cloud Run에 SSR 배포. 처음으로 48,706개 아파트 매물이 검색엔진에 색인 가능. 전 플랫폼 개발/운영 환경을 분리하여 팀이 두려움 없이 배포 가능.

2. **코드베이스 통합** (11월): 3개 프론트엔드 레포를 공유 패키지 기반 단일 모노레포로 통합. 이전에는 레포 간 조율이 필요했던 기능이 하나의 PR로 배포 가능. 이후 모든 것의 전제 조건 — 통합된 코드베이스 없이는 병렬 AI 에이전트가 충돌.

3. **테스트와 모니터링으로 안정화** (12월): 웹, 앱, 어드민 전반에 1,184개 테스트 구축. 프론트엔드와 서버리스 레이어에 Sentry 통합. v3.2.0을 4개 플랫폼에 동시 배포 — 팀이 처음으로 확신을 갖고 조율된 릴리스를 수행.

4. **사용자 경험 최적화** (1월): 안정적이고 통합된 코드베이스 위에서 공격적인 성능 최적화를 안전하게 수행. PageSpeed 20→80. 모바일 LCP 3.3초→1.3초. 외형적 개선이 아닌 — 48K+ 매물 페이지의 SEO 순위와 이탈률에 직접적 영향.

5. **운영 자동화** (2월): Sentry 에러 캠페인에서 수동 분석 작업의 규모가 드러남. Sentry 웹훅을 수신하여 4개 레포에 걸쳐 이슈를 진단하고, 수정 PR을 생성하며, 텔레그램에 보고하는 자율 봇을 구축 — 온콜 엔지니어의 필요성을 대체.

각 단계는 이전 단계가 있어야만 가능했습니다. 분산된 코드베이스는 최적화할 수 없습니다. 환경 분리 없는 플랫폼은 테스트할 수 없습니다. 모니터링 없이는 분석을 자동화할 수 없습니다. 12개월의 로드맵은 프로젝트 목록이 아닌 — 각 결정이 이전 모든 결정의 가치를 증폭시키는 **복리 투자의 연쇄**였습니다.
